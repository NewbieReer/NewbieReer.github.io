前言

​ 本文参考资料几乎全部来自《Windows PE权威指南》，加上自己的一些理解和整理。 尽管本文对大致的资源结构有了介绍，但是并没有涉猎对于某资源具体地解析和修改。如果之后博主需要用到这些没有涉猎的知识，或者时间充裕的情况下，会继续写文章介绍。

​ 如果读者只想了解【资源表】相关知识，可以直接从资源表的组织方式开始看起。

资源分类

​ 资源指的是windows的各种界面，包括：加速键[1]、位图、光标等。
​ 上述总共16个标准资源，但是在这16种资源之中，微软为开发者留了一个扩展，也就是第10项标准资源类别【RT_RCDATA】。后来对于新引入的资源（非标准资源），微软对其数据的组织方式不再明示。因此很多第三方软件，不得不扩展【RT_RCDATA】这个类别，将其所有新定义的类型全设置为RT_RCDATA。

程序常用的六类资源

位图、光标、图标资源

一般这几个资源对应的有ico、cur[4]、ani[5]、bmp文件。

这三种资源在脚本文件[6]中的定义格式如下：

  • 位图：nameID BITMAP [DISCARDABLE] 位图文件名
  • 光标：nameID CURSOR [DISCARDABLE] 光标文件名
  • 图标：nameID ICON [DISCARDABLE] 图标文件名

nameID

资源的名字，程序需要资源的时候需要用到它，类似文件的句柄

BITMAP CURSOR ICON

资源类型

DISCRADBLE

可选项，意思是不用的时候可以从内存暂时卸载掉。

例

  • TITLE_ICON icon “abc.ico”
  • 1000 icon discardable 123.ico
  • ICO_MAIN ICON “D:\masm32\source\chapter7\main.ico”

资源表的组织方式

资源的组织方式和文件有点类似，从根目录开始，下设：一级子目录、二级子目录以及三级子目录。

一级子目录

按资源类型分类，如【光标】一级子目录、【位图】一级子目录、【菜单】一级子目录、【字符串】一级子目录等。

二级子目录

按照【资源ID】分类。比如，在一个菜单的一级子目录下，其下可以有：

  • IDM_OPEN的ID号为2001
  • IDM_EXIT的ID号为2002
  • IDM_1的ID号为4000

等多个菜单项。

三级子目录

按照资源的【代码页】分类。即不同【语言】代码页对应不同的数据。其中根据语言可以分为：简体中文、英文、繁体中文等多个代码页。

上面【资源目录结构单元】分为一个【资源目录（头）】和若干个【资源目录项】。其实他们是一个个的结构体，下面我们来逐一介绍。

资源目录头

Characteristics

资源属性，保留为将来使用，须为0。

TimeDataStamp

时间戳，创建资源的时间。

MajorVersion、MinorVersion

资源版本号，【未用】，大部分情况下为0。

【!】NumberOfNameEntries

以名称[2]命名的资源个数。

【!】NumberOfIdEntries

以ID[3]命名的资源个数。

所以上面提到的“若干个【资源目录项】”，以及上图的【a+b】个资源目录项，其实指的是【NumberOfNameEntries】+【NumberOfIdEntries】个资源目录项。

资源目录项

由上述可知，一个资源目录拥有【NumberOfNameEntries】+【NumberOfIdEntries】个资源目录项（对应下图的m+n）。

资源目录项的结构体，关于其中某些字段涉及的c语言知识，后面会给出解释

该结构总共有两个union，长度都是DWORD，而且在不同级别目录下这几个字段有不同含义：

第一个union

  • 如果最高位【NameIsString】为1，表示低地址【NameOffset】是一个指向Unicode字符串的指针。
    • 第一层：非标准类型。NameOffset指向非标准类型的名字，比如前面举例中提到的【DLLTYPE】。
    • 第二层：非标准命名。低31位指向【非标准的类型下】的命名，类似于前面举例提到的【DIB_WINRESULT】
    • 第三层：非标准的语言。即无预定义的代码页，低31位指向非标准的语言的名字。
  • 如果最高位【NameIsString】为0，表示该字段为一个编号。
    • 第一层：标准类型。低16位组成整数标识符ID，如若该值为【03h】，则名字表示预定义当中的【ICON】。
    • 第二层：标准命名。低16位组成整数标识符ID定义名字。
    • 第三层：标准的语言。低16位组成整数标识符ID，如ID=2052，表示该语言为Simpled_Chinese。

第二个union

  • 如果最高位【DataIsDirectory】为1，低位数据指向下一级目录块的地址。
    • 第一层：低31位组成整数偏移地址，偏移指向下一级目录。
    • 第二层：低31位组成整数偏移，偏移指向下一级目录。
    • 第三层：【DataIsDirectory】不为1。毕竟没有下一级目录了。
  • 如果最高位【DataIsDirectory】为0，低位数据指向描述资源数据块的指针，通常出现在第三级目录中。
    • 第一层：【DataIsDirectory】不为0。
    • 第二层：【DataIsDirectory】不为0。
    • 第三层：表示该字段指向一个IMAGE_RESOURCE_DATA_ENTRY。

注意上述资源目录项中的偏移，指的是基于【资源起始位置】的偏移

资源数据项

资源数据项的结构体

这个资源数据项是通过三次目录定位之后才找到的（见资源目录项第二个union）

【!】OffsetToData

一个【RVA】值，注意文件访问的时候转化为文件偏移。这里指向的资源不是赤裸裸的资源信息，而是附加了一些结构数据的资源块。

【!】Size1

资源数据的大小。

CodePage

代码页【未用】，大多情况下为0。

Reserved

保留字段，总是为0。

RVA和RAW的转化

​ 也即目标数据在【内存中的偏移】和在【文件中的偏移】的转化。
​ RAW - PointerToRawData = RVA - VirtualAddress
​ RAW = RVA - VirtualAddress + PointerToRawData
​ 需要注意的是【PointerToRawData】以及【VirtualAddress】，在上式中表示的是在【IMAGE_SECTION_HEADER结构体】里的字段，指的是【磁盘文件中节区起始地址】以及【内存中节区起始地址】。

IMAGE_IMPORT_DESCRIPTOR

​ 首先介绍IMAGE_IMPORT_DESCRIPTOR（简称【IID】）。输入表由【IID数组】开始，以一个【全0】的IID结构结束。

每个被PE【隐式链接[1]】的DLL都有一个IID结构。

【INT数组】以一个内容为【0】的【IMAGE_THUNK_DATA】结构结束。

TimeDataStamp字段

一个32位的时间标志，可以忽略

ForwarderChain字段

这是第1个被转向的API索引，一般为0，在程序引用一个DLL的API，而这个API又在引用其他的DLL的API时使用（但这样的情况很少出现）。

【!】Name字段

DLL名字指针，比如”KERNEL32.DLL” “USER32.DLL”。字符串以00结尾。

【!】FirstThunk字段

【IAT】的【RVA】。【IAT】是一个【IMAGE_THUNK_DATA】结构体数组[3]。

示意图

OriginalFirstThunk和FirstThunk字段相似，指向了同一种结构体数组【IMAGE_THUNK_DATA】。这些数组的叫法INT：Import Name Table 输入名称表，IAT：Import Address Table 输入地址表。下图[4]展示的是一个可执行文件正在从USER32.DLL里输入一些API。

读者看到这里可能会产生疑惑，为什么INT和IAT指向的是【同一个地方】，不要着急，下面会有解释。

关于IMAGE_THUNK_DATA

  • 当IAMGE_THUNK_DATA[6]最高位为【1】时，表示函数以【序列号】方式输入，这时剩下的低位[5]被看作一个函数序号。
  • 当IMAGE_THUNK_DATA字段最高位为【0】时，表示函数以【字符串】类型的函数名方式输入，这时此结构体是一个地址，指向一个【IMAGE_IMPORT_BY_NAME】结构。

关于IMAGE_IMPORT_BY_NAME

Hint字段

记录[8]本函数在DLL的输出表的【序号】。该域被PE装载器用来在DLL的输出表里快速查询函数。非必须，一些链接器将它设置为0。

Name字段

输入函数的【函数名】。一个ASCII字符串，NULL结尾。[9]

输入地址表

为什么【INT】和【IAT】会一起指向【IMAGE_IMPORT-BY_NAME】结构：

  • OriginalFirstThunk，指向INT，【不可改写】
  • FirstThunk，指向IAT，由PE装载器【重写】

PE加载器先搜索【OriginalFirstThunk】指向的【INT】指向的【IMAGE_IMPORT_BY_NAME】结构指向的输入函数的【地址】。
然后加载器用函数真正的【入口地址】来代替【FirstThunk】指向的【IAT】中的元素。[10]
上图已经转换成了下图的样子，这时输入表的其他部分就不重要了，程序依靠IAT提供的函数地址就可以正常运行。

两种特殊情况：

  • 一些函数仅由序号引出。也就是说不能通过函数名调用它们，只能通过它们的位置调用。
  • 程序OrginalFirstThunk值为0。初始化时，系统根据FirstThunk的值找到【指向函数名】的地址数组。再根据函数名得到【入口地址】，再用入口地址【取代地址数组】的原值。
